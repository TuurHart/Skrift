Same tiers, same colors, same option sets as the desktop mocks (naming-review.html opt-out model + resolver-inline.html / name-unlink.html popover idiom) — re-skinned for a small screen and finger targets. The interaction model is carried verbatim: one link per person at the first mention; opt-out auto-link for distinctive names; tan-dotted suggestions for common-word / ambiguous names that need one tap; tapping a link offers Unlink / Change / Open with an Undo toast. Build notes — each screen → its iOS control

This is the phone half of the name-linking surface. It binds to the same shared Sanitiser / NamesData (Skrift_Native/Shared/Naming/) the Mac uses, so a link the phone shows and a link the Mac writes are byte-identical. The phone keeps Memo.transcript RAW and re-derives the styled tiers on demand via MemoLinking — a names edit shows up immediately, never stale, and the upload contract (phone sends RAW, Mac links) is untouched.

1 · Memo detail body An attributed Text (or a thin NSTextView-backed UIViewRepresentable for tappable ranges) rendered from MemoLinking.attributed(raw, names). Three run styles map 1:1 to the tier colors: .foregroundColor(.link #9d8ff7) for LINKED, .foregroundColor(.suggest #bda481) + dotted .underlineStyle for SUGGESTED, no attributes for PLAIN, .foregroundColor(.t2) for .repeat. Tap on a linked/suggested run opens the matching sheet; only the first mention per person is interactive (later ones render .repeat).
2 · Which-person chooser A .confirmationDialog (or a small custom anchored popover) headed by the single naming-review line Which Jack? · 2 people share this name, with that mock's exact option set: one button per candidate (Jack Hutton / Jack Tanner, the selected one trailing a ✓), ＋ New person… (pushes the Person editor pre-filled), Leave as plain text. The footer states the signed-off whole-note semantics: pick once → applies to the note; a rarer second Jack is tapped + resolved separately (no per-occurrence resolver).
3 · Linked-name actions A .confirmationDialog with the green “✓ this note is about / Hendri van Niekerk” header and three actions: Unlink — just a side-mention (demotes to a dotted suggestion, re-promotable; restores the alias AS SPOKEN), Change person… (reopens the chooser), Open Hendri’s note (jumps to the person in Names / Obsidian). The action posts an Undo toast — a transient overlay holding the pre-action linked-flag snapshot, exactly like name-unlink.html. This matches naming-review's signed-off 3-action set (Unlink / Change / Open — no “Unlink all”, no Cancel); the name-unlink “Unlink all mentions in this note” variant is only adopted if we switch to that idiom.
4 · People-in-this-note chips Reference only. This is the OLDER opt-in chip bar — the signed-off opt-out model (naming-review.html, 2026-06-16) DELETED it and moved interaction into the prose (screens 1–3). Kept here because the user asked to see it: a horizontal wrap of toggle chips above the body (✓ Full Name = linked/on, ＋ Name = candidate, dashed ＋ Someone else…). If we ever want a roster glance, this is the component; default IA is prose-only.
5 · Person editor A Form reached two ways — New person… from a chooser (pre-filled with the tapped word) OR a row tap in Settings → Names. Four field blocks mirror the Mac: Full name (canonical / link target), Aliases (token field + the live “saying ‘Bruno’ → Bruno Aragorn” demo line), Short name (the [[Bruno Aragorn|Bruno]] display alias), Voice (the “〰 Voice enrolled” status row — label only, no help line, mirroring the Mac editor). Saving a new person re-scans the open note so they appear as a suggestion/chip immediately. Edits sync via the existing NamesData LWW (union voiceEmbeddings) to all devices + the Mac.

What is faithfully matched from the Mac

  • Tier colors are exact: LINKED #9d8ff7 (the newest signed-off naming-review tier), SUGGESTED text #bda481 over a 1.6px dotted #ab9676, ambiguous variant = resolver-inline .amb verbatim: plain --t1 #e4e4e7 text + faint rgb(124 107 245/.13) bg + 1.5px purple dotted underline + the accent superscript ?. PLAIN is --t1 #e4e4e7; .repeat (a 2nd mention of a linked person) is --t2 #8b8b97.
  • Popover anatomy is the live idiom: #1d1f29 bg, 11px radius, the rotated-45° caret pointing at the tapped word, 18px accent-tinted avatar/icon circles, a .5px separator dividing primary from secondary actions, the 10px --t3 footer hint with a leading info/chevron glyph. Same component reused across screens 2 & 3.
  • Option sets are verbatim: the which-person chooser is naming-review.html's (single Which Jack? · … header, Hutton / Tanner / ＋ New person… / Leave as plain text); the prune/fix actions + the unlink sub-line (“→ plain ‘Hendri’, as spoken”) come from naming-review.html + name-unlink.html.
  • Person editor reproduces the four labeled blocks, the help lines (Full name / Aliases / Short name only — Voice is a status row, no help line, per the Mac), the alias recognition demo line (resolved name in accent), and the green “voice enrolled” status — the Mac’s “two doors, one editor.”
  • Prose typography holds the Mac’s review feel on a phone column: 21px/700/-.02em title, 16px / 1.72 body, the shared SF system font stack.

Decisions to weigh in on

  • Tappable text vs. confirmationDialog. Screens 2 & 3 are drawn as anchored popovers (to keep the green header + rich rows). A plain .confirmationDialog is more “stock iOS” but drops the rich header. Pick: custom anchored popover (matches the Mac, more build) vs. confirmationDialog (cheaper, less faithful).
  • Chip bar in or out (screen 4). The opt-out model says prose-only. Confirm we ship without the chip bar (screens 1–3 are the real UI), or keep it as an optional roster header.
  • Whole-note resolution (locked). Per the signed-off naming-review opt-out model, picking a person in screen 2 applies to the whole note; a rarer second Jack is tapped and resolved separately (no per-occurrence resolver). The screen-2 footer now states this; resolved away from resolver-inline’s per-occurrence promise toward the signed-off spec.
